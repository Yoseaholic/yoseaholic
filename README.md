## 📖 一、项目简介

**MOBILEMOREAGENT** 是一套基于多模态大语言模型（Multimodal LLM）构建的移动设备智能自动化操控框架。该系统的核心思路是：**通过一台电脑，借助 ADB（Android Debug Bridge）协议远程操控一台 Android 手机**，让 AI 代替人类完成各种复杂的手机操作任务。

与传统的按键精灵、Appium、Auto.js 等脚本化自动化工具不同，MOBILEMOREAGENT **不需要提前编写任何针对特定应用的操作脚本**，也不需要了解目标应用的内部结构。用户只需要用自然语言描述想要完成的任务（例如"帮我打开微信，给张三发一条消息说明天下午三点开会"），系统就能自主地感知屏幕、理解界面、规划步骤、生成代码、执行操作，并在每一步操作后进行智能验证，最终完成整个任务闭环。

### 🎯 这个项目能做什么？

想象一下以下场景：

- 🛒 **自动购物**：你告诉系统"帮我在淘宝上搜索蓝牙耳机，按销量排序，找一个100元以下的加入购物车"，系统会自动完成从打开淘宝、搜索商品、切换排序方式、筛选价格到加入购物车的全部操作。
- 📧 **自动发送消息**：你说"在微信里给妈妈发一条消息说今晚回家吃饭"，系统会自动打开微信、找到联系人、输入文字并发送。
- 🗺️ **自动导航**：你说"用高德地图查一下从公司到机场怎么走"，系统会自动打开高德地图、输入起点终点并展示路线。
- 📱 **自动设置**：你说"帮我把手机的Wi-Fi关掉，蓝牙打开"，系统会自动进入设置完成操作。
- 🧪 **自动化测试**：QA 工程师只需用自然语言描述测试用例，系统就能像真实用户一样操作应用并验证结果。

**简而言之，任何你能在手机上手动完成的操作，MOBILEMOREAGENT 都有能力通过 AI 自动帮你完成。**

## 🧠 二、核心设计理念

MOBILEMOREAGENT 的核心设计理念是**模拟人类操作手机时的认知过程**。当一个人操作手机时，实际上在不断重复以下思维循环：

> **看屏幕 → 理解当前状态 → 思考下一步该做什么 → 动手操作 → 看操作结果 → 判断是否成功 → 决定继续还是结束**

MOBILEMOREAGENT 将这一人类认知过程精确地映射为一套 AI 驱动的技术流程：

| 人类行为 | 系统对应模块 | 技术实现 |
|:--------:|:------------:|:--------:|
| 👀 看屏幕 | 视觉感知层 | ADB 截屏 + OCR + 图标检测 |
| 🧠 理解当前状态 | 屏幕语义解析 | 多模态模型理解界面结构 |
| 🤔 思考下一步 | 认知决策层 | 思考模型（Reasoning Model）进行任务拆解 |
| 🖐️ 动手操作 | 动作执行层 | 代码模型生成 ADB 命令并执行 |
| 👀 看操作结果 | 视觉验证层 | 再次截屏 + 识图模型比对 |
| ✅ 判断是否成功 | 状态验证层 | 识图模型判断预期变化是否发生 |
| 🔄 继续/结束 | 闭环控制器 | 检测下一目标元素，决定循环或退出 |

### 3.1 🔌 设备交互层（Device Interaction Layer）

> **对应文件**：`MobileAgentE/controller.py`
> **职责**：作为系统与 Android 设备之间的通信桥梁，负责所有底层硬件交互。

设备交互层是整个系统的"手脚"，它封装了完整的 ADB 命令行工具调用逻辑，向上层提供简洁统一的 Python API。`controller.py` 内部实现了以下关键功能：

#### 3.1.1 设备连接与初始化

系统启动时，`controller.py` 会自动检测通过 USB 数据线或 Wi-Fi 连接的 Android 设备。检测过程包括：

- 执行 `adb devices` 获取已连接设备列表
- 对目标设备执行 `adb shell wm size` 获取**屏幕物理分辨率**（如 1080×2400）
- 执行 `adb shell wm density` 获取**屏幕像素密度**（DPI）
- 执行 `adb shell getprop ro.build.version.release` 获取 **Android 系统版本号**
- 这些元数据信息对于后续的坐标映射和适配至关重要

#### 3.1.2 屏幕截图捕获

通过 `adb shell screencap -p` 命令实时捕获设备屏幕的 PNG 截图，并通过 `adb pull` 传输到本地计算机上。这张截图将作为视觉感知层的输入源。为了保证实时性，系统对截图流程进行了优化，单次截图耗时通常控制在 200-500ms 以内。

#### 3.1.3 触控与输入模拟

`controller.py` 支持模拟以下所有人类可在手机上执行的物理操作：

| 操作类型 | ADB 命令示例 | 应用场景 |
|:--------:|:------------:|:--------:|
| 单击（Tap） | `adb shell input tap x y` | 点击按钮、选择选项 |
| 长按（Long Press） | `adb shell input swipe x y x y 1500` | 触发长按菜单 |
| 滑动（Swipe） | `adb shell input swipe x1 y1 x2 y2 duration` | 翻页、滑动列表 |
| 文本输入（Type） | `adb shell input text "hello"` | 在输入框中键入文字 |
| 按键模拟（Key Event） | `adb shell input keyevent KEYCODE_BACK` | 返回、Home、音量键 |
| 回车确认 | `adb shell input keyevent 66` | 搜索确认、表单提交 |

所有坐标参数均由视觉感知层提供的矢量坐标经过分辨率换算后得到。例如，若视觉层返回某按钮的归一化坐标为 `(0.5, 0.3)`，设备分辨率为 `1080×2400`，则实际点击坐标为 `(540, 720)`。

#### 3.1.4 设备状态监控

`controller.py` 还负责监控设备的运行时状态，包括当前前台应用包名（`adb shell dumpsys window | grep mCurrentFocus`）、电池电量、网络连接状态等。这些信息可用于异常场景的判断与处理。

### 3.2 👁️ 视觉感知层（Visual Perception Layer）

> **对应文件**：`MobileAgentE/text_localization.py`、`MobileAgentE/icon_localization.py`、`MobileAgentE/crop.py`
> **职责**：对屏幕截图进行全方位的视觉解析，提取所有可交互元素的语义信息与精确坐标。

视觉感知层是系统的"眼睛"。它接收设备交互层传来的屏幕截图，输出一份结构化的**屏幕状态报告（Screen State Report）**，告诉认知决策层"屏幕上有什么、在哪里、是什么意思"。

#### 3.2.1 📝 文本定位模块（text_localization.py）

文本定位模块是视觉感知层中最核心的组件。它采用高精度的 OCR（光学字符识别）引擎对屏幕截图进行全图扫描，提取所有可见的文本内容。但与普通 OCR 不同的是，该模块不仅识别文本内容，还会返回每个文本块的**精确像素坐标**，并将其转换为**归一化矢量坐标**。

具体来说，对于每一个检测到的文本元素，模块会输出以下信息：

```json
{
  "text": "搜索",
  "confidence": 0.97,
  "bbox": {
    "x_min": 0.35,
    "y_min": 0.08,
    "x_max": 0.65,
    "y_max": 0.12
  },
  "center": {
    "x": 0.50,
    "y": 0.10
  }
}

其中：

text：识别出的文本内容（如"搜索"、"确认"、"下一步"、"登录"等）
confidence：识别置信度（0-1 之间，越接近 1 表示越确定）
bbox：文本边界框（Bounding Box）的四个角坐标，使用归一化矢量坐标表示
center：边界框中心点的归一化坐标——这就是后续点击操作所需的目标坐标
为什么要使用矢量坐标（归一化坐标）？

不同手机的屏幕分辨率差异巨大（从 720P 到 4K），如果直接使用像素坐标，同一套逻辑在不同设备上将完全失效。通过将坐标归一化到 [0, 1] 区间，系统实现了跨分辨率的坐标兼容。实际执行时再由 controller.py 根据当前设备的分辨率进行反向映射，计算出真实像素坐标。

3.2.2 🖼️ 图标定位模块（icon_localization.py）
并非所有可交互的 UI 元素都包含文字。许多按钮是纯图标形式的（如"返回"箭头、"分享"图标、"购物车"图标、"+"号按钮等）。对于这类元素，OCR 完全无能为力。

图标定位模块专门解决这一问题。它利用计算机视觉中的目标检测算法或模板匹配技术，识别屏幕中的图形化 UI 元素，并为每个检测到的图标提供：

图标的语义类别（如 "back_button"、"search_icon"、"close_button" 等）
图标的中心点坐标（同样为归一化矢量坐标）
检测置信度
通过文本定位模块与图标定位模块的协同工作，系统能够构建出一张完整的屏幕交互元素地图，覆盖了屏幕上所有可被用户点击、输入或滑动的区域。

3.2.3 ✂️ 图像裁剪预处理模块（crop.py）
在某些场景下，全屏截图的信息量过大，会干扰模型的判断。例如，当需要精确识别某个下拉菜单中的选项时，菜单区域可能只占屏幕面积的 20%，其余 80% 的信息对当前任务而言都是噪声。

crop.py 提供了智能的感兴趣区域（Region of Interest, ROI）裁剪功能。它根据当前任务的上下文信息，自动确定需要重点关注的屏幕区域，对截图进行裁剪、放大和增强处理，然后再送入 OCR 或识图模型，显著提升了小目标检测和密集文本区域场景下的识别准确率。

3.3 🧠 认知决策层（Cognitive Decision Layer）
对应文件：MobileAgentE/agents.py、MobileAgentE/api.py、MobileAgentE/chat.py
职责：接收用户指令与屏幕状态，完成任务理解、步骤拆解、代码生成与执行验证。

认知决策层是 MOBILEMOREAGENT 的"大脑"，也是多模态大模型能力集中发挥作用的核心层级。该层内部调用了三种不同专长的 AI 模型，各司其职、协同工作：

3.3.1 🤔 思考模型（Reasoning Model）—— 任务拆解引擎
模型角色：规划者（Planner）
实现位置：agents.py

当用户发来一个自然语言任务指令后，思考模型首先对指令进行深度理解。它会结合当前手机屏幕的状态（由视觉感知层提供的屏幕状态报告），运用链式思考（Chain-of-Thought, CoT） 推理方式，将一个宏观的、模糊的用户目标逐步拆解为一系列有序的、可原子化执行的子任务。

举例说明：

用户输入："帮我在大众点评上找一家附近评分4.5以上的火锅店，看看营业时间。"

思考模型的拆解结果：

子任务1: 在手机桌面上找到并打开"大众点评"应用
子任务2: 点击搜索框，输入关键词"火锅"
子任务3: 点击搜索按钮或回车确认
子任务4: 在搜索结果页面中，查找筛选/排序功能
子任务5: 设置筛选条件：评分 ≥ 4.5，距离按"最近"排序
子任务6: 从筛选后的列表中选择第一家符合条件的火锅店
子任务7: 进入店铺详情页，找到并查看"营业时间"信息
子任务8: 将营业时间信息截图或记录后，返回报告给用户
思考模型还会分析各子任务之间的前后依赖关系。例如，子任务3必须在子任务2完成后才能执行，子任务5需要在子任务4找到筛选入口后才能操作。这些依赖关系会被编码为一个有序任务队列，指导后续的逐步执行。

同时，对于中国特有的应用（如微信、支付宝、淘宝、抖音等），系统还会参考 data/Mobile-Eval-E/custom_tips_for_cn_apps.txt 中预定义的应用专属操作提示词，帮助思考模型更好地理解国产应用的特殊 UI 布局和交互习惯。

3.3.2 💻 代码模型（Code Generation Model）—— ADB 命令生成器
模型角色：执行者（Executor）
实现位置：api.py

代码模型是一个经过特殊训练或提示工程优化的大语言模型，专门用于将自然语言描述的操作意图转化为精确的、可直接执行的 ADB 命令代码。

它接收的输入包括：

当前要执行的子任务描述（来自思考模型）
目标元素的名称和坐标（来自视觉感知层）
设备分辨率信息（来自设备交互层）
生成示例：


输入：
  子任务描述: "点击屏幕上的'搜索'按钮"
  目标元素: { text: "搜索", center: {x: 0.50, y: 0.10} }
  设备分辨率: 1080 × 2400

输出（生成的 ADB 命令）：
  adb shell input tap 540 240

输入：
  子任务描述: "在搜索框中输入'火锅'"
  目标元素: { text: "搜索框", center: {x: 0.50, y: 0.08} }
  设备分辨率: 1080 × 2400

输出（生成的 ADB 命令序列）：
  adb shell input tap 540 192        # 先点击搜索框使其获得焦点
  adb shell input text '火锅'        # 输入中文文本
  adb shell input keyevent 66        # 按下回车键确认搜索
代码模型的优势在于它能理解操作意图的上下文。例如，它知道在输入文本之前需要先点击输入框使其获得焦点，在搜索完成后需要按下回车键确认——这些隐含的操作步骤，传统脚本需要人工显式编写，而代码模型能够自动推断并补全。

3.3.3 👁️‍🗨️ 识图模型（Visual Understanding Model）—— 操作验证器
模型角色：验证者（Verifier）
实现位置：agents.py + api.py

识图模型是一个具备视觉理解能力的多模态大模型。它在每次操作执行后被调用，负责对比操作前后的屏幕截图，完成以下三项核心判断：

判断一：操作是否成功执行？

模型会分析操作前后的屏幕差异。例如：

如果上一步是"点击搜索框"，那么操作后屏幕上应该出现光标闪烁或软键盘弹出
如果上一步是"输入'火锅'并搜索"，那么操作后应该出现搜索结果列表
如果屏幕没有任何变化，则判定操作可能失败
判断二：是否存在下一阶段任务需要点击的目标元素？

模型会扫描当前屏幕，检测子任务队列中下一个任务所需的目标元素是否已经出现在屏幕上。例如：

如果下一个子任务是"点击'筛选'按钮"，模型会在当前屏幕上搜索是否存在"筛选"二字或对应的图标
如果找到了，则返回该元素的名称和位置坐标，指导下一步操作
如果没找到，可能需要滑动页面、等待加载或进行异常处理
判断三：所有任务是否已经完成？

当任务队列中的最后一个子任务执行完毕后，识图模型会进行最终确认。如果确认任务目标已达成（如营业时间信息已显示在屏幕上），则向系统发出任务完成信号。

3.3.4 💬 对话管理模块（chat.py）
chat.py 负责维护整个任务执行过程中的对话上下文。由于大模型的上下文窗口长度有限，当操作步骤较多时（如超过 20 步），早期的对话记录可能超出模型的处理范围。chat.py 实现了上下文压缩与滑动窗口机制，在保留关键历史信息的同时，控制每次模型调用的 token 消耗量，确保长序列任务也能稳定运行。

3.4 🖥️ 用户接口层（User Interface Layer）
对应文件：ui.py、run.py、inference_agent_E.py、static/
职责：提供人机交互界面，支持任务输入、过程可视化与结果输出。

3.4.1 Web 可视化控制面板（ui.py + static/）
ui.py 基于 Python Web 框架构建了一个轻量级的可视化控制面板，前端使用了以下技术栈：

Bulma CSS（static/css/bulma.min.css）：现代化的 CSS 框架，提供美观的响应式布局
FontAwesome（static/css/fontawesome.all.min.css）：丰富的矢量图标库
Bulma Carousel/Slider（static/js/bulma-carousel.js、static/js/bulma-slider.js）：用于截图轮播和进度展示
自定义交互逻辑（static/js/index.js）：处理前后端实时通信
通过 Web 面板，用户可以：

📋 输入任务：在文本框中用自然语言描述想要完成的手机操作
📺 实时观看：在浏览器中实时查看手机屏幕的镜像画面
📊 跟踪进度：查看当前正在执行的子任务、已完成的子任务列表和整体进度条
📝 查看思考日志：查看 AI 模型的推理过程——它在"想"什么、为什么这样操作
🔴 紧急中止：在任何时刻一键中止任务执行
3.4.2 程序入口文件

⭐ 六、技术亮点
1️⃣ 多模态融合感知 —— 看得"全"
传统自动化工具要么只依赖 OCR（无法处理图标按钮），要么只依赖 Accessibility 服务（部分应用不支持）。MOBILEMOREAGENT 同时融合了 OCR 文本识别和图标目标检测两条感知管线，实现了对移动端 UI 界面的全元素覆盖——无论目标是文字标签、图形按钮还是混合型控件，系统都能精准定位。

2️⃣ 矢量坐标自适应 —— 用得"准"
所有视觉坐标均经过分辨率归一化处理（转换为 0-1 范围的矢量坐标），再由 controller.py 根据实际设备分辨率进行动态反算。这意味着同一套操作逻辑可以在 720P 低端机、1080P 中端机、2K/4K 旗舰机乃至各种尺寸的模拟器上无缝运行，无需任何适配修改。

3️⃣ 三模型协同架构 —— 想得"深"
系统创新性地引入了三个专业化 AI 模型的协同工作模式：

模型	专长	职责
🤔 思考模型	逻辑推理与规划	理解用户意图，将复杂任务拆解为可执行的子任务序列
💻 代码模型	代码生成	将自然语言操作意图转化为精确的 ADB 命令
👁️ 识图模型	视觉理解	验证操作结果，检测下一步目标元素
三个模型各有侧重、互相补充，避免了单一模型"既要思考又要写代码还要看图"所导致的能力稀释和错误累积。

4️⃣ 操作后验证机制 —— 做得"稳"
这是 MOBILEMOREAGENT 区别于其他 Agent 框架的最关键特性。系统不是盲目地按预定脚本执行操作，而是在每一步操作之后都会：

📸 重新截屏
🔍 调用识图模型分析操作前后的界面差异
✅ 判断操作是否成功执行
🎯 检测下一步需要交互的目标元素是否已出现
🔄 根据判断结果决定继续、重试或退出
这一机制使系统具备了自我纠错能力，有效防止了传统自动化中"一步错、步步错"的级联故障，将长序列任务（20+ 步操作）的成功率提升到了实用水平。

5️⃣ 智能退出策略 —— 收得"好"
当识图模型判断所有子任务均已完成，且当前屏幕上不存在下一阶段需要点击的目标元素时，系统不会无限循环或异常退出，而是会优雅地终止任务：

向操作模块 controller.py 发送退出指令
根据配置执行返回桌面（KEYCODE_HOME）或关闭当前应用（KEYCODE_BACK）
生成完整的任务执行报告
通过 Web 界面或 API 返回给用户

❓ 八、常见问题
Q1: 支持哪些手机品牌？
A: 支持所有开启了 USB 调试（Developer Options → USB Debugging）的 Android 设备，包括但不限于小米、华为（需安装 HDB 驱动）、OPPO、vivo、三星、一加、Google Pixel 等。同时也支持各类 Android 模拟器（如雷电模拟器、夜神模拟器、Android Studio 自带 AVD 等）。

Q2: 需要 Root 权限吗？
A: 不需要。所有操作均通过标准 ADB 协议完成，不涉及 Root 权限。

Q3: 能操作所有应用吗？
A: 理论上可以操作任何在屏幕上可见的应用界面。但部分应用可能有防自动化检测机制（如支付密码输入、银行类应用的安全键盘），这类场景下系统的操控能力可能受限。

Q4: 任务执行失败怎么办？
A: 系统内置了多级容错机制：

操作级重试：单步操作失败时自动重试最多 3 次
步骤级重规划：如果某个子任务反复失败，思考模型会尝试寻找替代操作路径
任务级中止：当确认无法完成时，系统会安全退出并向用户报告失败原因和已完成的步骤
Q5: 对网络有什么要求？
A: 由于系统需要调用云端大模型 API（思考模型、代码模型、识图模型），因此运行系统的电脑需要有稳定的互联网连接。手机端无特殊网络要求（除非任务本身涉及网络操作）。
